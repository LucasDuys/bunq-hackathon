# Scale Tradeoffs: Full Context vs RAG vs Hierarchical Summary vs Lazy Retrieval

## Summary
There is no single correct way to feed campaign data to an LLM agent — the right approach depends on dataset size, query type, and how often data changes. This document makes a concrete recommendation for each scale tier and explains exactly when to switch strategies.

## Key Concepts
- **Full context load**: Serialise all bricks into the prompt before every LLM call
- **RAG (Retrieval Augmented Generation)**: Embed bricks as vectors, retrieve the most relevant ones per query using semantic similarity
- **Hierarchical summary**: Pre-compute a compact summary of the structure, load full detail on demand via tools
- **Lazy retrieval**: Load nothing upfront; the agent fetches what it needs turn by turn via tool calls
- **pgvector**: PostgreSQL extension for storing and querying vector embeddings — uses existing infra
- **Pinecone / Weaviate**: Dedicated vector databases — higher throughput, more operational overhead
- **Embedding**: Converting text to a numerical vector so similarity can be computed mathematically

## How It Works

### Approach 1: Full Context Load

```typescript
// Current implementation in Agent.service.ts
const campaignString = await AgentService.getCampaign(accountId, userId, id, title);
// campaignString is the full hierarchy passed to every LLM call
```

**How it works**: All bricks are serialised and included in the system or user message on every turn. The LLM has complete visibility into the campaign.

**Token cost for a 200-brick campaign**: ~20,000 tokens per call × $3/MTok = $0.06 per turn. At 100 turns/day = $6/day.

### Approach 2: RAG with Vector Search

```typescript
import { OpenAIEmbeddings } from "@langchain/openai";
import { PGVectorStore } from "@langchain/community/vectorstores/pgvector";

// Index bricks (run when campaign changes)
const embeddings = new OpenAIEmbeddings({ model: "text-embedding-3-small" });
const vectorStore = await PGVectorStore.fromDocuments(
    bricks.map(b => new Document({
        pageContent: `${b.title} ${b.subType} ${JSON.stringify(b.data)}`,
        metadata: { brickId: b.id, campaignId: b.campaignId }
    })),
    embeddings,
    { postgresConnectionOptions: { connectionString: process.env.MYSQL_CONNECTION_STRING } }
);

// At query time: retrieve top-k relevant bricks
const relevantBricks = await vectorStore.similaritySearch(userQuery, 5);
// Pass only these 5 bricks to the LLM instead of all 200
```

**How it works**: Bricks are pre-embedded. At query time, the user's message is embedded and the most similar bricks are retrieved. Only those ~5 bricks are passed to the LLM.

Source: https://js.langchain.com/docs/integrations/vectorstores/pgvector

### Approach 3: Hierarchical Summary + On-Demand Detail

```typescript
// Summary: passed upfront (cheap, ~500 tokens)
const summary = `Campaign: Air France Spring 2026
- Meta Campaign 1 (id: abc) [budget: €5000]
  - Ad Set 1: Retargeting NL (id: def)
    - Ad 1: Homepage Hero (id: ghi)
    - Ad 2: Product Banner (id: jkl)
  - Ad Set 2: Prospecting EU (id: mno)`;

// Detail: fetched by the agent when needed
const get_brick = tool({
    execute: async ({ brickId }) => fullBrickDetails(brickId)
});
```

**How it works**: The compact hierarchy string (already generated by `getCampaign()`) is the only upfront context. When the agent needs a field value, it calls `get_brick`. This is the pull model vs. the push model of full context.

---

### Approach 4: Lazy Retrieval (Fully Tool-Driven)

```typescript
// No campaign data in the initial context at all
const systemPrompt = `You are the brick editor. The user has campaign ID ${campaignId} open.
Use get_campaign to view the structure, then get_brick to fetch specific bricks.`;

// Agent calls tools for everything
const get_campaign = tool({ execute: () => getCampaignSummary(campaignId) });
const get_brick = tool({ execute: ({ id }) => getBrickById(id) });
```

**How it works**: The agent starts cold and pulls everything it needs. Maximum flexibility, but adds latency for every new piece of information needed.

### Vector Database Comparison

| Database | Hosting | Cost | Latency | Best for |
|----------|---------|------|---------|---------|
| pgvector | Self-hosted (extend existing PG/MySQL) | ~$0 extra infra | 5-50ms | Small-medium scale, existing SQL infra |
| MongoDB Atlas Vector Search | Already in use | Included in Atlas tier | 10-50ms | Reuse existing MongoDB |
| Pinecone | Managed SaaS | Free tier → $70+/month | 5-20ms | High scale, simple ops |
| Weaviate | Self-hosted or managed | Free OSS / $25+/month | 10-30ms | Multi-modal, GraphQL queries |
| Chroma | Self-hosted | Free | 1-10ms local | Development / prototyping |

**Project note**: The project already uses MongoDB. MongoDB Atlas Vector Search requires zero new infrastructure — use it if RAG is needed.

Source: https://www.mongodb.com/products/platform/atlas-vector-search

## Use Cases Relevant to This Project

| Scale | Typical scenario | Recommended approach |
|-------|-----------------|---------------------|
| <50 bricks | Single campaign, early stage | Full context load |
| 50-200 bricks | Typical production campaign | Hierarchical summary + tools (current) |
| 200-500 bricks | Large multi-brand campaign | Hierarchical summary + metadata-filtered tool fetching |
| 500-2,000 bricks | Enterprise account with many campaigns | Lazy retrieval + pgvector/MongoDB for search |
| >2,000 bricks | Theoretical ceiling, bulk operations | RAG + map-reduce for analysis |

## Tradeoffs

### Accuracy
- **Full context** wins on accuracy — the model sees everything
- **RAG** loses on structured relational queries ("show all children of ad group 2") because vector similarity doesn't understand parent-child relationships well
- **Hierarchical summary** is accurate for navigation; loses on fine field values until `get_brick` is called
- **Lazy retrieval** is accurate but requires the agent to know what to ask for

### Latency
| Approach | Added latency | Why |
|----------|--------------|-----|
| Full context | 0ms setup, but longer LLM input processing | More tokens = slower TTFT |
| RAG | +50-200ms per query for embedding + search | Network call to vector DB |
| Hierarchical summary | 0ms (already computed) | Same as full context minus most data |
| Lazy retrieval | +100-500ms per tool call | Each `get_brick` is a DB round-trip |

### Cost
| Approach | Cost per turn (200 bricks) | Cost per turn (2,000 bricks) |
|----------|--------------------------|------------------------------|
| Full context | ~$0.06 | ~$0.60 (or impossible) |
| RAG (5 bricks retrieved) | ~$0.002 | ~$0.002 |
| Hierarchical summary | ~$0.003 | ~$0.003 |
| Lazy retrieval | ~$0.001 + tool calls | ~$0.001 + tool calls |

### Operational Complexity
- **Full context**: Zero — just serialise and send
- **Hierarchical summary**: Low — `getCampaign()` already does this
- **RAG**: Medium — requires embedding pipeline, index updates on brick mutations, vector DB ops
- **Lazy retrieval**: Low to medium — requires good tool design but no new infrastructure

### Data Freshness
- **Full context**: Always fresh (loaded at query time)
- **RAG**: Stale if index not updated on brick edits — this is the main operational risk for mutable data
- **Hierarchical summary**: Fresh for structure; `get_brick` is always fresh
- **Lazy retrieval**: Always fresh (pulled live)

## Recommended Approach for This Project

**Now (0-200 bricks)**: The current hierarchical summary approach (`getCampaign()` hierarchy string + `get_brick` tool) is correct. No change needed to the loading strategy.

**Optimisations to make immediately**:
1. Strip audit fields from the hierarchy string to reduce token count by ~30%
2. Add `cache_control` to cache the campaign summary across turns in the same conversation

**At 200-1,000 bricks**: Switch to lazy retrieval — remove the upfront `getCampaign()` call from the context and make the agent call `get_campaign` as its first tool use. The tool already exists. This scales infinitely.

**Do not implement RAG** unless the use case becomes: "find bricks matching a semantic description" (e.g., "show me all bricks about summer promotions"). For structural navigation and targeted edits, hierarchical summary + tools is always more accurate and simpler than RAG on relational data.

**If RAG is ever needed**: Use MongoDB Atlas Vector Search (already in the stack) rather than adding Pinecone or Weaviate. The operational simplicity of not adding a new service outweighs any performance difference at this scale.
