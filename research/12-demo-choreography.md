# 12 — Demo choreography

## Problem
The pitch is a 3-minute demo to a room that sees 20 hackathon submissions in an afternoon. Sequencing and pacing decide whether "carbon accounting" sticks or slides off.

## The one-liner
"We turn bunq transactions into a monthly, audit-ready carbon report — and automatically fund your EU carbon reserve."

## The 3-minute sequence
1. **Setup (15s)** — show the overview page, 61 tx already streaming, €39k spend, no CO₂e yet. "This is Acme BV. Their bunq business account just… works." Point at the `ingested via webhook` stat.
2. **Kick off close (10s)** — click **Run Carbon Close**. Page navigates to `/close/[id]` with the pipeline animating. "Once a month, the agent closes the books."
3. **Show clusters + questions (45s)** — pipeline pauses at `AWAITING_ANSWERS`. Read out the first question: "Amazon — €4,180 — was that electronics, office supplies, or resale goods?" Click **electronics**. Repeat for AWS (`compute_eu`) and Google Cloud. Confidence bar animates 31% → 46%. "Two minutes of human time per month. No receipts."
4. **Show proposed actions (20s)** — scroll to **Proposed actions**. "€275 into the Carbon Reserve — below the €500 policy threshold so no approval needed. Plus three EU credit purchases: biochar, reforestation, peatland." Highlight the EU flag and the "≥70% removal" policy.
5. **Approve (10s)** — click **Approve & execute**. State goes to COMPLETED. "In production that's a live bunq intra-user transfer. In sandbox the bunq app would show the sub-account balance increase."
6. **Show CSRD report (30s)** — navigate to `/report/2026-04`. "Narrative generated by Sonnet. E1-6 table — spend and CO₂e by category. E1-7 — credit mix with project type, registry, EUR. Methodology footnote with data lineage. This is CSRD ESRS E1 disclosure territory."
7. **Show ledger (20s)** — `/ledger`. "Every state change, every external call, every user decision is in a SHA-256 hash chain. The badge in the top-right verifies the chain. Auditors can replay our entire reasoning path."
8. **Impact matrix + close (20s)** — `/categories`. "And this is what we tell the company next month: train instead of short-haul flight is 16× lower. Plant-based catering 3× lower. The agent's real value isn't the reserve — it's the behaviour change that shrinks next month's bill."

## What to rehearse
- **Refresh timing** after answering each question — the page uses Next.js router.refresh() which takes 100-300ms. Practice so it looks intentional.
- **Reset before demo** — `pnpm reset` wipes the DB, re-seeds, starts fresh. Run twice back-to-back to confirm determinism.
- **Mock vs live** — for in-room demo, `ANTHROPIC_MOCK=1 BUNQ_MOCK=1` is safer (no API dependence). Flip to live only if you can tolerate a 5-second Sonnet call live.

## Failure modes to prepare for
- **Dev server holds stale DB handle** after `pnpm reset`. Solution: restart dev server post-reset. Build a `reset-demo-restart` script.
- **First question generation is slow** (Sonnet cold-start, mock is instant) — rehearse with live mode if you plan to use it.
- **Bunq sandbox outage** — `BUNQ_MOCK=1` so the demo works without the real API.
- **Presenter forgets the Amazon line** — it's the emotional hook (€4k of ambiguity, two questions, resolved). Do not skip.

## Cut list under time pressure
In order of what to drop if the demo is running long:
1. Step 8 (Impact matrix) — ~20s savings.
2. Step 7 (ledger demo) — ~20s savings.
3. Step 6 (CSRD report) — ~30s savings. **Keep if any one must stay; it's the "audit-ready" proof.**

## Decisions for this build
- Mock-first defaults so the demo never depends on external APIs.
- `scripts/reset-demo.ts` deterministic reseed.
- `StartCloseButton` routes to `/close/[id]` on success so presenter doesn't need to copy URLs.
- Keep the approve button obvious — big, green, one click.
